#### UART
- 每次收到任何 byte 的时候，产生 receive interrupt
    read - consoleread(manage cons.buf) - current process wait for a full line
    shell 进程等待指令 read 的时候，就已经睡眠了；后续等待完整的一行之后再被kernel唤醒
    此时，kernel 通过 uartintr && consoleintr 处理输入字符
- 每次完成send a byte的时候，产生 transmit complte interrupt
    first byte: write - uartputc - uartstart(start to transmit)
    remaining : uartintr - uartstart(from interrupt generated by arrival of previous transmit arrive)
    以上实现了，通过 interrupt 和 buffer 对进程活动和硬件活动的解耦

## Chapter 6 - Locking
并发问题来源：多核处理器并行、线程切换、中断

#### __sync_synchronize() 有什么用
当以多线程进行运行的时候，需要考虑防止重排的问题。
考虑单线程情况下，指令顺序优化、编译器优化等都会根据后续内容是否以来前序指令，如果依赖则不会调整次序，从而保证 as-if-serial。
但是多线程情况下，尽管单一线程能不会出现这样的问题，但是在多线程情况下，指令优化、编译器无法判断当前指令的依赖关系，例如线程 A 中分别修改了data、flag，对于 A 来说，这两者是独立的；但是对于线程 B 来说，可能是基于 flag 的值，来处理 A 的内容，所以也具有次序依赖关系。因此，需要人为的通过显式编程的方式 __sync_synchronize() 在 data、flag 赋值之间指明，防止指令重排

#### how to use locks
There are a few basic principles. 
First, any time a variable can be written by one CPU at the same time that another CPU can read or write it, a lock should be used to keep the two operations from overlapping. 
Second, remember that locks protect invariants: if an invariant involves multiple memory locations, typically all of them need to be protected by a single lock to ensure the invariant is maintained.

The need for a global lock acquisition order means that locks are effectively part of each function’s specification ——> lock-order chains，为了完成某项操作，按照特定的先后顺序来获取了一组锁，重点在于获取和一组锁。释放的顺序没有关系，因为对Lock A、B，假设两个CPU都是以 A-B的顺序获取锁，假设先释放了A，第二个进程会阻塞在 acquire(&lockB)的地方；而如果先释放B，由于A没有释放，另一个CPU也会阻塞在 acquire(&lockA)的地方

#### Locks and interrupt handlers
deadlock 的链路在于，当一个 CPU acquire one lock 之后，此时，interrupt 跳转到handler，然而，handler 中也需要去acquire 同一个 lock，此时发生死锁。CPU 原指令等待interrupt 返回，但是interrupt 此时又在等 lock release。

因此，xv6 中的方案是，acquire lock 之后到 release 之前，关闭当前 CPU 的interrupt，避免 acquire 之后 interrupt 出现竞态；但是如果 interrupt acquire 的lock 需要 wait 某个lock，而这个lock 现在被其他CPU 占用，这是可以的

#### 为什么 acquire、release 内部需要 push_off、pop_off
假设一个链路上分别 acquire 了 Lock A、B、C，那么在 release Lock C的时候，就不能去直接 enable interrupt，否则依旧会导致上述问题，此时，A、B都还没有释放，interrupt 中可能有需要 acquire 这两个 lock，导致死锁

#### xv6 的两种锁：自旋锁、休眠锁
##### 自旋锁
- 持锁进程不允许系统中断，也不允许主动 yield，直到锁释放。
- 对于前者：如果 CPU 持锁的时候，此时发生中断，切换到 handler，而 handler 中也希望 acquire 同一个锁。然而，因为进入中断之后，不再接受任何硬件中断，但是内部又需要去 acquire lock，导致不会返回。而 CPU 又需要等待中断返回才能继续运行，bomb！因此，如果一个锁在 CPU 和 interrupt handler 中都会使用，那要确保中断开始时，这个锁是free的（xv6 更保守，任何lock 持有的过程中，不允许中断。保证interrupt 发生时，所有 lock 都是 free的） ——> CPU wait handler to finish， handler wait for CPU's lock
- 对于后者，一个进程持有锁之后 yield，此时另一个进程也 acquire 这个锁，但是 acquire 之后又关闭了中断，导致当前进程永远不会被 timer interrupt yield，从而使得前一个进程永远无法被 device timer 唤醒，bomb！ ——> Proc A wait to be awakened, Proc B wait for A's lock and disable interruption

存在的问题
-   持锁过程中当前进程关闭中断，也不允许 yield，意味着时间片和进程本身都无法对进程进行调度。（影响当前 CPU 的其他进程
    Another drawback of spinlocks is that a process cannot yield the CPU while retaining a spinlock
    注意：同一 CPU 一个进程 acquire lock 之后，不可能有其他进程再同去 acquire 同一个spinlock，因为acquire的时候就屏蔽了中断，在释放之前，不会被timer 切到其他进程
-  对其他 acquire 同个lock的进程造成浪费。（影响其他 CPU 的进程
    Holding a spinlock that long would lead to waste if another process wanted to acquire it, since the acquiring process would waste CPU for a long time while spinning. 
    CPU 1 中 acquire 了 Lock A，CPU 2 上的内容也想去 acquire，这时候就会发生 waste

##### 休眠锁
- 持锁进程允许中断、休眠，但是中断 handler 中不允许使用 sleeplock
    中断必须返回，否则，中断向量始终阻塞，否则handler 中通过sleeplock yield 之后不会再被唤醒
- acquiresleep 会 yield CPU，所以休眠锁也不能在自旋锁临界区里使用
- 但是反过来，自旋锁可以在休眠锁的临界区中使用

存在的问题：
-- 获取、释放锁的开销很大，涉及到进程调度，速度慢

综上，自旋锁适用于耗时短的临界区；休眠锁适合耗时冗长的指令
