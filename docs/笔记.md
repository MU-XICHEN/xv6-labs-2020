## Chapter 7 - Scheduling


### Scheduling
p->lock 的 acquire 和 release 在swtch 调度场景下，会存在 Thread A(user process) acquire 但在 Thread B(scheduler)中 release 的情况，用于保持进程状态的不变性  
参考 yield 代码，如果在 swtch 之前就释放锁，会导致期间另一个CPU 可以 acquire p->lock 并切换该进程的内核栈，从而导致两个 CPU 运行在了相同 stack 上  

#### coroutines
sched and scheduler are co-routines of each other  
两个函数并不是在一次调用中直接返回，而是切换到另一个函数的上下文执行。  
区别于函数调用，更类似于函数A 执行前一半，然后执行函数B 的后一半，互相成为彼此的前后部分。

#### p->lock 保护的invariants(不变性)
- running 的进程要保证，timer interrupt 可以安全的从该进程上切换走  
此时，CPU 的寄存器保证了进程寄存器的值；以及 cpu->proc 指向该进程
- runnable 的进程要保证，scheduler 可以安全的运行它  
此时，p->context 保存这进程的寄存器内容，且没有 CPU 执行在该进程的kernel stack 上，以及没有 ->proc 指向该进程
而在 swtch 时，proc 的寄存器内容时局部变化的，因此需要持有 p->lock 来保护，以防 CPU 访问不完整的 proc 内容

#### xv6 是如何让进程访问 cpuid 的？
多核情况下，会存在两个 scheduler 运行，不同 CPU 也拥有属于自己的寄存器，tp 就是其中一个  
tp 在内核态和用户态时的定义不同，在用户态这用于指示正在运行的线程指针 thread pointer
然后在内核态，tp 被视为存储当前 cpu 的 hartid

tp 寄存器的更新路径为：
- 在 boot 阶段，每个 CPU 都会给自己的 tp 设置为 hartid
- 之后在每个 CPU 通过 usertrapret 调度进入用户进程时，会将tp的内容存入 trapframe->kernel_hartid
- 用户进程执行... 
- 发生 timer interrupt
- 进入内核，恢复 tp，之后被设置为 RUNNABLE
- 再次被两个 CPU scheduler 其中之一 调度
- 此时 usertrapret 中的 r_tp()，返回的就是当前 CPU的 tp了 
（在哪个 CPU 上被唤醒，r_tp()因此读取谁的tp值）

#### semaphore - V(producer) && P(consumer)
xv6 kernel 中对 sleep 中的调用都是通过 consumer 的方式来进行的，通过 while 来循环等待 consume  
以下实现保证了 invariants  
如果 V 先执行，条件改变；P 执行时，条件为 false，直接break  
如果 P 先执行，在 while 之前 acquire 确保原子性，这期间 V 的执行都会 wait s->lock；sleep 内部确保睡眠之前安全地 release s->lock
```c
// semaphore - V(producer) && P(consumer)
void V(struct semaphore *s)
{
    acquire(&s->lock);
    s->count += 1; // modify conditions
    wakeup(s);
    release(&s->lock);
}

void P(struct semaphore *s)
 {
    acquire(&s->lock);
    while(s->count == 0) // conditions
        sleep(s, &s->lock);
    s->count -= 1;
    release(&s->lock);
 }
```

#### sleep and wakeup 内部实现
sle ep 中存在对于两个 lock 的 acquire和 release  
lk 在进入 sleep 前，caller 要确保 acquire，从而保证 wakeup 会等待 sleep 释放了这个锁之后调用  
然后 sleep 内部，在 acquire p->lock 之后，可以安全地 release lk，因为此时 wakeup 内部会 acquire p->lock，继续等待  
sleep 在修改进程状态，并 sched 之后，释放 p->lock  
wakeup 最终获得 p->lock 之后，继续执行，从而使得进程 RUNNABLE  
PS: 一些场景下，多个进程可能会在同一个 channel(chan) 上 sleep，while 的作用就用于解决该问题，当任意一个进程 woken 并消费之后，其他的进程需要继续进行 sleep（被唤醒但是没有内容，进入下一个 while）  
**综上，lk 确保即使 V 在 P 之前执行，即当执行到 P 时不会进入 sleep，可以得到正确的状态；同时，一旦进入了 sleep，说明此时 wakeup 并没有执行，因此要保证，在 release lk 并进入 SLEEPING 之前，通过 acquire p->lock 可以阻塞一切 wakeup 调用（传接力棒相当于），最终进程 sleep 之后得以被 wakeup 的执行进行唤醒**
