#### Xv6 如何处理 traps:
- hardware actions taken by the RISC-V CPU
    1. If the trap is a device interrupt, and the sstatus SIE bit is clear, don’t do any of the
    following.
    2. Disable interrupts by clearing SIE.
    3. Copy the pc to sepc.
    4. Save the current mode (user or supervisor) in the SPP bit in sstatus.
    5. Set scause to reflect the trap’s cause.
    6. Set the mode to supervisor.
    7. Copy stvec to the pc.
    8. Start executing at the new pc.
    ps: 提升状态，跳转到指定 vec，注意此时页表还没有切换
- an assembly “vector” that prepares the way for kernel C code
    保存寄存器，恢复页表，跳转 trap handler
- a C trap handler that decides what to do with the trap
    决断是 interrupt、exception、syscall
- the system call or device-driver service routine.d
    specific routine

#### traps 发生的三种场景：
*注意：kernel 在处理 interrupt 的过程中，会阻塞后续的 interrupt*
- traps from user space
    trap 发生时，硬件处理并没有切换页表，意味着 uservec 在用户空间地址中必须有映射，使得 trap 发生时，可以根据 sepc 获得对应的物理地址；同时，为了使得 uservec 在内核空间地址中可以被往下执行，意味着，uservec 在kernel 和 user 空间中必须映射在同一片物理内存.
    因此，当即将执行用户代码时，stvec 会被设置为 uservec，用来处理用户态下，trap 发生时如何响应
- traps from kernel space
    注意 kernel 本身没有进程的 trapframe 这样的结构可以存储栈帧，kernelvec 将栈帧内容存储在栈上
- timer interrupts  

#### trampoline 被映射在相同的内核和用户虚拟内存的意义？
the trampoline page 被映射在用户虚拟空间和内核虚拟空间的相同虚拟地址，因此在切换页表之后，uservec 中的指令可以继续往下执行。例如，Program counter 指向一个指令地址，那这个指令，在切换页表之后仍需要可以正常访问

#### COW 写时复制机制
fork()      瞬间仅复制页表——不复制物理内存父子进程共享物理页，权限设为只读
仅读取时     双方继续共享——运行速度极快，无需额外内存占用
一旦写入     硬件触发 Trap（缺页异常）——内核申请新物理页，拷贝数据，更新页表
写入完成     后两个进程拥有各自独立的物理页——互不干扰，数据隔离完成
ps：最初双方维护自己的页表，共享内存；后续触发修改后，针对被修改的页，重新分配内存给进程，并更新其 pagetable 指向新的物理页


#### alarm 传递的 handler，不能直接在 kernel 中直接执行
- 安全问题，可能存在恶意代码
- 地址引用问题，即使可以通过 pt 获得handler 的pa，但是其内部访问的内容还是用户空间地址，而trap此时还使用的kernel pt，不存在地址映射
