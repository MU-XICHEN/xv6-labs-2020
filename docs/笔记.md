microkernel kernel vs monolithic kernel

进程最重要的内核状态是其页表、内核栈、运行状态

每个进程有两个 stack，一个用户栈、一个内核栈。执行用户指令的时候，使用用户栈，中断或者系统调用的时候在内核栈上执行

系统调用之后，硬件提升特权等级，然后修改 PC 指向内核定义的一个程序入口点。系统调用完成之后，通过调用 sret 来回到用户空间，同时降低特权等级

xv6 会让分页硬件使用当前运行进程的用户空间中的 p->pagetable ；同时页表也用于记录 进程内存的物理页的地址

mret 从 M-Mode 进入 S-Mode

硬件提供定时器中断，进入内核（操作系统），操作系统进行进程调度（时间片概念）


硬件启动后，ROM 加载 xv6 kernel 到指定位置，进行一些初始化（例如，构建初始stack，以运行 C 语言，配置以使得 mret 调用可以从 M-Mode 进入 S-Mode，设置定时中断等），之后在S-Mode 情况下，初始化第一个进程（shell，设置 RUNABLE），使其等待调度

*trapframe 和 trampoline*

    trapframe = 用户态执行状态的完整快照

    trampoline 做的事：
    1️⃣ 用户 → 内核
        保存寄存器到 trapframe
        切换页表
        进入内核 trap handler
    2️⃣ 内核 → 用户
        从 trapframe 恢复寄存器
        切换回用户页表
        执行 sret 回到用户态

用户态和内核态的页表是独立的
- 用户态和内核态使用的页表是不一样的，所有进程的内核区都处在虚拟空间的高地址，根据独立的页表映射到相同的物理内存。因此，用户态的指令通过系统调用传递给内核态的地址是无法直接被内核使用的
- 所以内核中通过基于用户进程自身的页表来访问特定的地址，从而获取资源(参考 fetchStr)。同时，会判断用户希望访问地址是否有效（在进程的虚拟地址空间内等），避免访问了未分配的虚拟地址空间（取决于用户本身代码段、数据段的大小，以及 malloc 动态分配、动态链接映射的共享库空间等）
